● コードを読んだ結果、インジェクション攻撃は成立しません。理由を説明します。                                                                    
                                                            
  処理フローの確認                                                                                                                              
                                                                                                                                                
  lex_line → parse_pipeline → expand_pipeline → exec_pipeline                                                                                   
                                                                                                                                                
  展開（expand）はパース後に行われます。expand_pipeline が cmd->argv[]                                                                          
  を展開済み文字列で埋める時点では、すでにパイプラインのAST構造は確定しています。                                                               
                                                                                                                                                
  具体的なケース                                            

  export VAR="; rm -rf /"                                                                                                                       
  echo $VAR
                                                                                                                                                
  1. lex_line: echo と $VAR の2トークンに分割                                                                                                   
  2. parse_pipeline: cmd = {argv: ["echo", "$VAR"]} として確定
  3. expand_pipeline → expand_argv_word → expand_word: $VAR を文字列 "; rm -rf /" に置換し argv[1] に格納                                       
  4. exec_pipeline: execve("echo", ["echo", "; rm -rf /"], env) を呼ぶ                                                                          
                                                                                                                                                
  expand_word が返すのは ただの文字列です。その結果を lex_line や parse_pipeline に再度通すコードはありません。; や |                           
  は展開後のargvの中では単なる文字でしかなく、シェルの演算子として解釈されません。
                                                                                                                                                
  念のため他のリスク面も                                                                                                                        
  
  ┌────────────────────────┬─────────┬────────────────────────────────────────────┐                                                             
  │          経路          │ 安全か  │                    理由                    │
  ├────────────────────────┼─────────┼────────────────────────────────────────────┤                                                             
  │ $VAR → argv            │ ✅ 安全 │ parse後に展開、再lexしない                 │
  ├────────────────────────┼─────────┼────────────────────────────────────────────┤
  │ $VAR → heredocデリミタ │ ✅ 安全 │ strip_quotes のみ（$ 展開なし）            │                                                             
  ├────────────────────────┼─────────┼────────────────────────────────────────────┤                                                             
  │ $VAR → リダイレクト先  │ ✅ 安全 │ ファイル名として使うだけ（コード実行なし） │                                                             
  ├────────────────────────┼─────────┼────────────────────────────────────────────┤                                                             
  │ eval                   │ ✅ 安全 │ そもそも eval が実装されていない           │
  └────────────────────────┴─────────┴────────────────────────────────────────────┘

  歴史的にはbashが「環境変数に格納された関数定義を起動時に自動evalする」実装になっていたため脆弱でした。このminishellはそのような自動eval機構は

  結論: ご指摘の攻撃は、このアーキテクチャでは成立しません。展開結果がパーサに戻らない設計が保護になっています。

executeとなっていればインジェクション可能だと思います。となると, SQLもbashとおなじアーキテクチャを取れば攻撃を防げるのではないでしょうか？      
                                                                                                                                                
● 非常に鋭い洞察です。その直感は正しく、実際にそれが解決策として実装されています。                                                              
                                         
  SQLインジェクションが起きる理由                                                                                                               
   
  古典的な脆弱なコード：                                                                                                                        
                                                            
  query = "SELECT * FROM users WHERE name = '" + user_input + "'"                                                                               
  db.execute(query)  # ここで初めてlex/parse
                                                                                                                                                
  これはまさに 「データを埋め込んでからparse」 の順序です：                                                                                     
                                                                                                                                                
  文字列連結（expand相当） → lex/parse → execute                                                                                                
                                                            
  user_input = "' OR '1'='1" を入れると、パーサには：                                                                                           
  SELECT * FROM users WHERE name = '' OR '1'='1'            
  という「正当なSQL構文」が渡ってしまう。                                                                                                       
                                                            
  Prepared Statement は bash と同じアーキテクチャ                                                                                               
                                                                                                                                                
  stmt = db.prepare("SELECT * FROM users WHERE name = ?")                                                                                       
  stmt.execute(user_input)                                                                                                                      
                                                                                                                                                
  処理順序を見ると：
                                                                                                                                                
  lex/parse("SELECT ... WHERE name = ?")                    
      → AST確定（?はパラメータスロット）                                                                                                        
          → スロットにデータをバインド（expand相当）                                                                                            
              → execute                                                                                                                         
                                                                                                                                                
  ┌─────────────────────────────────┬────────────────────────────────────┐                                                                      
  │              bash               │       SQL Prepared Statement       │
  ├─────────────────────────────────┼────────────────────────────────────┤
  │ echo $VAR をlex/parse           │ SELECT ... WHERE name = ? をparse  │
  ├─────────────────────────────────┼────────────────────────────────────┤
  │ $VAR → パース済みのargvスロット │ ? → パース済みのパラメータスロット │                                                                      
  ├─────────────────────────────────┼────────────────────────────────────┤                                                                      
  │ expand_word がスロットを埋める  │ バインドがスロットを埋める         │                                                                      
  ├─────────────────────────────────┼────────────────────────────────────┤                                                                      
  │ 結果は再lexされない             │ 結果は再parseされない              │
  └─────────────────────────────────┴────────────────────────────────────┘                                                                      
                                                            
  ';DROP TABLE users-- をバインドしても、パーサはすでにAST確定済みなので文字列データとして扱われるだけです。                                    
                                                            
  なぜ完全には解決しないか                                                                                                                      
                                                            
  ただし、bash との類比で見えてくる限界もあります。                                                                                             
                                                            
  識別子（テーブル名・カラム名）はパラメータ化できない：                                                                                        
  -- これはできない                                         
  SELECT * FROM ? WHERE ? = ?                                                                                                                   
  --           ↑テーブル名はパラメータスロットに入れられない
                                                            
  SQLの文法上、識別子と値は構文的に別物であり、パーサが「ここは識別子が来るべき場所」と確定した後にデータをバインドする仕組みが識別子に対しては 
  言語仕様にない。これは bash で言うとコマンド名自体を変数にするケース（$CMD arg）に相当し、どんなシェルでも word splitting や globbing         
  の対象になります。                                                                                                                            
                                                                                                                                                
  本質的な構造：                                                                                                                                
   
                    ┌──────────────────────────────────┐                                                                                        
                    │  「コードとデータを同じ表現に混ぜる」 │
                    │  → parse前に結合 → injection可能  │                                                                                       
                    └──────────────────────────────────┘                                                                                        
                                                                                                                                                
                    ┌──────────────────────────────────┐                                                                                        
                    │  「構造を先に確定し、データを後から」 │
                    │  → parse後にバインド → injection不可│                                                                                     
                    └──────────────────────────────────┘    
                                                                                                                                                
  同じ原理は他にも現れます：                                                                                                                    
  - HTML/JS インジェクション（XSS）: 文字列連結でDOMを生成 →
  脆弱。テンプレートエンジンがDOMツリーをparse済みの構造として扱ってからデータを埋める → 安全                                                   
  - eval(): 文字列をコードとして再parseする → injection可能（bash でいう eval $VAR）         
                                                                                                                                                
  まさに「parseとデータの混入を分離する」という原則が、injection攻撃全般への根本的な対策になっています。
