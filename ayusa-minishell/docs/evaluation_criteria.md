# minishell システム評価観点

---

## 用語定義

### クラッシュ

**プログラムエラーに起因して、プロセスが意図しないシグナルによって強制終了されること。**

`exit()` による意図的な終了（正常・エラー問わず）はクラッシュではない。

具体的には以下のシグナルによる終了を指す：

| シグナル | 原因 |
|----------|------|
| `SIGSEGV` | 不正メモリアクセス（NULL 参照・範囲外アクセス・use-after-free・スタックオーバーフロー） |
| `SIGBUS` | アライメント違反なメモリアクセス |
| `SIGILL` | 不正な機械語命令の実行 |
| `SIGFPE` | ゼロ除算・浮動小数点例外 |
| `SIGABRT` | `abort()` 呼び出し・`assert()` 失敗 |

---

## 評価観点

### 1. 仕様適合性

参照仕様（bash）に対してどれだけ正確に動作するか。

- 各コマンドの終了ステータスが bash と一致するか
- クォート・変数展開・ワイルドカード展開の規則が仕様通りか
- `TK_REDIR_RDWR`（`<>`）のように**トークンは生成されるが executor に処理がない機能**の存在
- SIGPIPE 処理のように**環境依存で bash と異なる挙動をする箇所**
  - bash は起動時に自プロセスの SIGPIPE を `SIG_IGN` に設定し、子プロセスには明示的に `SIG_DFL` にリセットして渡す
  - この minishell は `set_signal_child()` で SIGPIPE を設定しないため、起動元プロセスの SIGPIPE 設定をそのまま子プロセスに引き継ぐ
  - 結果：`nohup` 等の SIGPIPE=`SIG_IGN` 環境から起動した場合、`yes | head -1` のような処理で `yes` が終了せず無限ループになりうる

---

### 2. 堅牢性

正常系以外の入力・状態に対してクラッシュせず適切に振る舞うか。

**入力異常系**
- 空文字・スペースのみ・ネストしたクォート・不完全なリダイレクト等に対してクラッシュしないか
- 異常入力に対して適切なエラーメッセージを出力し、非ゼロ終了ステータスを返すか

**システムコール失敗への対処**
- `malloc` が `NULL` を返したとき、`fork` が `EAGAIN` を返したとき等にクラッシュせず適切に処理するか
- `open` 失敗・`pipe` 失敗・`execve` 失敗の各パスが正しくハンドルされるか

**動作環境の異常**
- `/tmp` 不在・容量不足、`/dev/urandom` 不在、fd 枯渇、メモリ不足の各状態で挙動が定義されているか

---

### 3. リソース管理

実行中・終了後にリソースが正しく解放されるか。

**メモリ**
- `malloc` で確保した領域がすべての実行パス（正常・エラー・シグナル割り込み）で `free` されるか
- 特に `cleanup_and_exit` を経由しない異常終了パスでのリークがないか

**ファイルディスクリプタ**
- `open` / `pipe` した fd が必ず `close` されるか
- `dup` / `dup2` で複製した fd（`saved_fds`）が `restore_fds` で確実に閉じられるか
- パイプの書き込み端・読み取り端が親プロセス側で `close` されるか（閉じ忘れると右側プロセスが EOF を受け取れない）

**プロセス**
- `fork` した子プロセスが必ず `waitpid` されるか（ゾンビプロセスが残らないか）
- heredoc の子プロセス・パイプの両子プロセス・サブシェルの子プロセス、すべて回収されるか

---

### 4. シグナル安全性

シグナルハンドラが安全に実装されているか。

**async-signal-safe**
- シグナルハンドラ内で呼ぶ関数が async-signal-safe であるか
  - `write()` は安全 ✓
  - `rl_on_new_line()`, `rl_replace_line()`, `rl_redisplay()` は async-signal-safe ではないが、`rl_catch_signals = 0` によりシグナルは readline の外で処理される前提の設計
- `g_sig` が `volatile sig_atomic_t` として宣言されており、シグナルハンドラとメインループ間で安全に共有されているか

**SIGPIPE**
- 自プロセスの SIGPIPE 設定が未定義（`SIG_DFL` のまま）であり、パイプの書き込み側になった場合に minishell プロセス自体が死ぬ可能性がある
- 子プロセスへの SIGPIPE 設定の引き継ぎが bash と異なる（上記「仕様適合性」参照）

---

### 5. セキュリティ

意図しない操作・権限昇格・情報漏洩が起きないか。

**一時ファイル**
- `/tmp/.minishell_heredoc_*` は `0600` で作成されており、他ユーザーによる読み取りを防いでいるか
- ファイル名の衝突チェック（`access(filename, F_OK) == -1` のループ）は行っているが、確認から作成までの間に別プロセスが同名ファイルを作れる TOCTOU 競合が残る

**環境変数・コマンド引数**
- 外部から与えられた環境変数やコマンド引数が意図しないコマンド実行につながらないか
- `execve` に渡す `envp` に内部管理変数（`?`）が含まれていないか（`builtin_env` では `?` を除外しているが `env_list_to_array` は除外していない）

---

### 6. パフォーマンス

応答性とリソース消費が実用的な範囲にあるか。

- コマンド実行のレイテンシ（`fork` + `execve` + `waitpid` のオーバーヘッド）
- 長大なパイプライン・多重 heredoc における fd 消費・プロセス数
- 大量ファイルが存在するディレクトリでのワイルドカード展開速度（`readdir` ループ + `malloc` 連鎖）

---

### 7. 移植性

異なる Linux 環境で一貫して動作するか。

- libc 実装（glibc / musl）の差異による `malloc` 挙動・`strerror` メッセージ文言の差
- GNU readline バージョン差による API 挙動の差
- ファイルシステム種類（ext4 / tmpfs / NFS）による `readdir` 順序の差
- ロケール・文字コード環境の差による日本語プロンプトの表示崩れ
- コンテナ / PID namespace 環境でのシグナル・プロセスグループの動作差

---

### 8. 保守性・テスト容易性

将来の変更・検証がしやすい構造になっているか。

- lexer / parser / expander / executor が適切に分離されているか（各コンポーネント単体でのテストが可能か）
- エラーパスが追いやすいか（エラー発生箇所から `cleanup_and_exit` までの経路が明確か）
- 未実装機能（`TK_REDIR_RDWR` 等）が明示的に管理されているか

---

## 観点まとめ

```
仕様適合性    ── bash との動作一致度・未実装・SIGPIPE 挙動差
堅牢性        ── 異常入力・syscall 失敗・環境異常への耐性（クラッシュしないこと）
リソース管理  ── メモリ・fd・プロセスの完全な解放
シグナル安全性── async-signal-safe・SIGPIPE の設定漏れ
セキュリティ  ── 一時ファイル競合・envp への ?変数混入
パフォーマンス── レイテンシ・スケーラビリティ
移植性        ── libc/readline バージョン・FS・ロケール・コンテナ
保守性        ── コンポーネント分離・テスト容易性
```
